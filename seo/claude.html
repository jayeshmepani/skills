🟡 Anti-Pattern Async Patterns

❌ Callback Hell / Old Promise Chains
fetch('/api/data')
  .then(r => r.json())
  .then(data => {
    fetch('/api/process', {
      method: 'POST',
      body: JSON.stringify(data)
    })
    .then(r => r.json())
    .then(result => {
      doSomething(result);
    })
    .catch(err => console.error(err));
  })
  .catch(err => console.error(err));
✅ async/await
async function processData() {
  try {
    const res = await fetch('/api/data');
    const data = await res.json();
    const result = await fetch('/api/process', {
      method: 'POST',
      body: JSON.stringify(data)
    }).then(r => r.json());
    doSomething(result);
  } catch (err) {
    console.error(err);
  }
}

ℹ
Core Rule: Use the element that most accurately describes the meaning of the content, not its appearance. Native elements come with built-in accessibility, keyboard support, and implicit ARIA roles — free of charge.

♿ ARIA & Accessibility

🚫
The #1 ARIA Rule: "No ARIA is better than bad ARIA." WebAIM found pages with ARIA averaged 41% more errors than those without. Always use native semantic HTML first. Only add ARIA when the native element cannot do the job.

The 5 Rules of ARIA

  • If you can use a native HTML element with the semantics you need, do so instead of adding ARIA
  • Do not change native semantics unless you really have to (don't add role="button" to a <h2>)
  • All interactive ARIA controls must be keyboard accessible (add tabindex="0" and key handlers)
  • Do not use role="presentation" or aria-hidden="true" on focusable elements
  • All interactive elements must have an accessible name (aria-label, aria-labelledby, or visible text)

🎨 Color Contrast — WCAG & APCA

WCAG 2.x (Current Legal Standard)

The WCAG 2.x contrast formula divides the relative luminance of the lighter color (L1) by the darker color (L2): CR = (L1 + 0.05) / (L2 + 0.05). The relative luminance of a color is computed by converting sRGB values to linear light and applying weights: L = 0.2126R + 0.7152G + 0.0722B (where each channel is linearized via gamma correction).

⚠
Exceptions to WCAG contrast: Decorative text, logotypes, disabled control text, and text in inactive UI components are exempt. However, placeholder text in inputs IS subject to contrast requirements.

APCA — Advanced Perceptual Contrast Algorithm (WCAG 3 Future)

APCA is a more sophisticated model that accounts for font size, weight, and polarity (dark-on-light vs light-on-dark). Scores are expressed as Lc (Lightness Contrast) from 0–106. APCA is directional — foreground and background order matters.

APCA Lc ValueMinimum Use
Lc 90+Preferred for fluent body text (14px/400 weight or larger)
Lc 75Minimum for columns of body text (18px/400 or larger)
Lc 60Minimum for regular-weight text at comfortable reading sizes
Lc 45Minimum for large bold text or non-text elements
Lc 30Incidental text or placeholder
Lc 15Point of near-invisibility for many users — avoid
ℹ
Why APCA? WCAG 2.x math can be misleading — e.g., orange text on white "passes" 4.5:1 but users find it difficult to read, while certain dark-mode combinations "fail" but are actually comfortable. APCA accounts for human perception including font weight and size. WCAG 3 is expected to adopt APCA. For now, ensure WCAG 2.x compliance and check with APCA tools for design quality.

Contrast Checklist

  • Body text: minimum 4.5:1 (WCAG AA), aim for 7:1 (AAA)
  • Large text (18px normal or 14px bold+): minimum 3:1
  • UI components and graphical objects: minimum 3:1
  • Focus indicators: minimum 3:1 against adjacent colors (WCAG 2.2 SC 2.4.11)
  • Placeholder text: minimum 4.5:1 (not exempt)
  • Error states: sufficient contrast for both color and the indicator shape
  • Don't rely on color alone — use icons, patterns, or labels too
  • Test in both light and dark mode
  • Check on gray/muted displays (simulating low-quality monitors)

📦 JSON-LD Structured Data

JSON-LD (JavaScript Object Notation for Linked Data) is Google's recommended format for structured data. It uses <script type="application/ld+json"> in the <head> or <body>. It enables rich results in search (star ratings, FAQs, recipes, breadcrumbs, etc.).

srcset & sizes Explained

srcset (width descriptors)

Lists available image files with their intrinsic widths (400w, 800w). The browser selects the best candidate based on sizes and screen density.

srcset (density descriptors)

Use 1x, 2x for fixed-size images that need high-DPI versions: srcset="logo.png 1x, logo@2x.png 2x"

sizes attribute

Tells the browser how wide the image will actually render in CSS. Use media conditions matching your CSS layout. The browser can then fetch the right image before CSS loads.

width & height attributes

Always include on <img>. Sets the aspect ratio in HTML, preventing Cumulative Layout Shift (CLS). CSS can still override visual dimensions.

⚡ Performance Best Practices

Critical Rendering Path

The browser must: fetch HTML → parse HTML → discover resources → fetch CSS/JS → parse CSS (CSSOM) → execute blocking JS → combine DOM + CSSOM into render tree → layout → paint. Every render-blocking resource delays this.

Resource TypeDefault BehaviorOptimization
HTMLParser-blocking when <script> encounteredUse defer or async on <script> tags
<link rel="stylesheet">Render-blocking — browser won't paint until CSS parsesInline critical CSS; load non-critical CSS async; use media attribute tricks
<script> in <head>Parser-blocking AND render-blockingAdd defer (executes after parse) or async (executes ASAP)
<script defer>Downloaded parallel; executes in order after HTML parsedUse for most scripts — maintains execution order
<script async>Downloaded parallel; executes immediately when ready (no order guarantee)Use for independent scripts (analytics, ads)
Web FontsCan cause FOIT (Flash of Invisible Text) or FOUTUse font-display: swap; preconnect to font CDN; self-host
ImagesNon-blocking but consume bandwidthLazy load, compress, use modern formats, set dimensions
 
<!-- Script loading patterns -->

<!-- ❌ Worst: render-blocking in head -->
<head>
  <script src="app.js"></script>
</head>

<!-- ✅ defer: most scripts; in head is fine -->
<head>
  <script src="app.js" defer></script>
  <script src="ui.js" defer></script>
  <!-- Both execute in order after parse, before DOMContentLoaded -->
</head>

<!-- ✅ async: independent 3rd-party scripts -->
<head>
  <script src="https://analytics.example.com/ga.js" async></script>
</head>

<!-- ✅ type="module": always deferred by default -->
<script type="module" src="main.js"></script>

<!-- Critical CSS: inline styles for above-the-fold -->
<style>
  /* Critical above-the-fold CSS here */
  body { margin: 0; font-family: system-ui; }
  .hero { min-height: 60vh; background: #000; }
</style>
<!-- Non-critical CSS: load async -->
<link rel="preload" href="styles.css" as="style"
      onload="this.onload=null;this.rel='stylesheet'">
<noscript><link rel="stylesheet" href="styles.css"></noscript>

🚀 Resource Hints

HintWhat It DoesPriorityUse WhenCaution
dns-prefetch Performs DNS resolution for cross-origin domain Low For domains you'll connect to but don't need immediately; fallback for browsers not supporting preconnect Only DNS — no TLS/TCP. Cost: basically free
preconnect DNS + TCP handshake + TLS negotiation for cross-origin Medium Domains for critical 3rd-party resources (fonts, CDN, APIs). Can reduce latency by 100-500ms Use sparingly — if connection unused within 10s (Chrome), browser closes it. Each TLS cert is ~3KB overhead
preload Downloads resource with high priority; doesn't execute High Critical resources the browser discovers late: fonts referenced in CSS, LCP images, key scripts NOT a hint — a mandatory instruction. Unused preloads show DevTools warning. Max 2-3 preloads

CSS & Font Optimization

/* font-display: controls text rendering during font load */
@font-face {
  font-family: 'MyFont';
  src: url('/fonts/myfont.woff2') format('woff2');
  font-weight: 400 900; /* variable font range */
  font-display: swap;  /* swap: shows fallback immediately, then swaps */
  /* optional: for non-critical fonts — shows nothing if slow */
  /* block: holds text 3s then shows (avoid) */
  /* fallback: 100ms blank, 3s swap, then stays fallback */
  unicode-range: U+0000-00FF; /* only load for these chars */
}

/* Content-visibility: skip rendering off-screen elements */
.off-screen-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px; /* estimated height to prevent layout shift */
}

/* will-change: hint browser to create compositor layer */
.animated-element {
  will-change: transform; /* use sparingly; GPU memory expensive */
}

/* Use transform/opacity for animations (not top/left/width) */
.slide-in {
  transform: translateX(0); /* ✅ GPU composited */
}
/* ❌ avoid: */
.slide-in-bad { left: 0; } /* triggers layout + paint */

JavaScript Lazy Loading with IntersectionObserver

// Lazy load images with IntersectionObserver
const imageObserver = new IntersectionObserver(
  (entries, observer) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        const img = entry.target;
        img.src = img.dataset.src;
        img.srcset = img.dataset.srcset || '';
        img.removeAttribute('data-src');
        observer.unobserve(img); // Stop observing once loaded
      }
    });
  },
  {
    rootMargin: '200px 0px', // Load 200px before visible
    threshold: 0.01
  }
);

document.querySelectorAll('img[data-src]').forEach(img => {
  imageObserver.observe(img);
});

// Lazy load components/sections
const sectionObserver = new IntersectionObserver(
  (entries) => {
    entries.forEach(async (entry) => {
      if (entry.isIntersecting) {
        const { default: module } = await import('./heavy-component.js');
        module.init(entry.target);
        sectionObserver.unobserve(entry.target);
      }
    });
  },
  { threshold: 0.1 }
);

document.querySelectorAll('[data-lazy-component]').forEach(el => {
  sectionObserver.observe(el);
});

👁 JavaScript Observer APIs

IntersectionObserver

Detect when an element enters/leaves the viewport or another element. Replace scroll event listeners for lazy loading, infinite scroll, animations on scroll.

ResizeObserver

Detect when an element's size changes. Replace window.resize for element-level responsiveness, component-based layouts.

MutationObserver

Detect DOM changes — attribute changes, child additions/removals. Replace polling, DOMSubtreeModified event (deprecated).

PerformanceObserver

Monitor performance entries (LCP, FID, CLS, resource timing). Essential for real user monitoring (RUM).

IntersectionObserver — Complete Reference

const observer = new IntersectionObserver(
  (entries, observer) => {
    entries.forEach(entry => {
      // entry.isIntersecting: boolean — in viewport?
      // entry.intersectionRatio: 0–1 — how much is visible?
      // entry.boundingClientRect: element dimensions
      // entry.rootBounds: root/viewport dimensions
      // entry.target: the observed element
      // entry.time: DOMHighResTimeStamp

      if (entry.isIntersecting) {
        entry.target.classList.add('visible');
      }
    });
  },
  {
    root: null,         // null = browser viewport
    rootMargin: '0px',  // expand/shrink intersection zone
    threshold: [0, 0.25, 0.5, 0.75, 1] // fire at these ratios
  }
);

observer.observe(element);
observer.unobserve(element); // stop watching specific element
observer.disconnect();        // stop all observations
observer.takeRecords();       // get pending entries

ResizeObserver

const ro = new ResizeObserver(entries => {
  for (const entry of entries) {
    const { width, height } = entry.contentRect;

    // Also available: entry.borderBoxSize, entry.contentBoxSize
    // entry.devicePixelContentBoxSize (physical pixels)

    element.style.setProperty('--width', `${width}px`);
  }
});

ro.observe(element, {
  box: 'border-box' // 'content-box' | 'border-box' | 'device-pixel-content-box'

ro.unobserve(element);
ro.disconnect();

  mutations.forEach(mutation => {
    if (mutation.type === 'childList') {

      // mutation.removedNodes: NodeList of removed nodes
    } else if (mutation.type === 'attributes') {

      // mutation.oldValue: previous value (if enabled)
    } else if (mutation.type === 'characterData') {
      // Text node content changed
    }

  attributeFilter: ['class'], // Only watch specific attributes
  attributeOldValue: true,   // Record old attribute values

📊 Core Web Vitals (2025)

LCP — Optimization Strategies

  • Use fetchpriority="high" and loading="eager" on the LCP image
  • Preload the LCP image with <link rel="preload" as="image">
  • Serve images in AVIF/WebP format; compress aggressively
  • Use a CDN to serve assets from edge locations
  • Avoid lazy-loading above-the-fold images
  • Eliminate render-blocking resources (defer/async scripts, critical-inline CSS)
  • Optimize TTFB with server-side rendering, caching, or edge computing
  • Never place an important image behind a CSS background — it's discovered later

INP — Optimization Strategies

  • Break up long tasks (>50ms) using scheduler.yield(), setTimeout(..., 0), or requestIdleCallback()
  • Defer non-critical JavaScript; avoid large JS bundles on the main thread
  • Use useTransition (React) or startTransition for non-urgent state updates
  • Minimize third-party scripts that block the main thread
  • Optimize event handlers — avoid heavy work synchronously in click/tap handlers
  • Use Web Workers for CPU-intensive tasks
  • INP measures ALL interactions, not just the first — a slow menu open hurts your score

CLS — Optimization Strategies

  • Always set width and height attributes on <img> and <video>
  • Reserve space for ads, embeds, and dynamic content with CSS min-height or aspect-ratio
  • Use font-display: swap with a well-matched fallback font to reduce FOUT shift
  • Don't insert content above existing content unless triggered by user interaction
  • Use CSS transform for animations instead of top/left/width changes
  • Use content-visibility: auto with contain-intrinsic-size to avoid scroll shifts
  • Animations that change layout properties (width, height, margin, padding) cause CLS
// Breaking up long tasks with scheduler.yield()
async function processLargeDataset(items) {
  for (let i = 0; i < items.length; i++) {
    processItem(items[i]);

    // Yield to browser every 50 items to remain responsive
    if (i % 50 === 0) {
      await scheduler.yield?.() ?? new Promise(r => setTimeout(r, 0));
    }
  }
}

✅ Master Pre-Launch Checklist

HTML Structure

  • <!DOCTYPE html> is first line
  • <html lang="..."> has correct language code
  • <meta charset="UTF-8"> within first 1024 bytes
  • <meta name="viewport" content="width=device-width, initial-scale=1.0"> present
  • No obsolete elements: <font>, <center>, <frame>, <marquee>, etc.
  • No obsolete attributes: type="text/javascript", align, bgcolor, border on img, etc.
  • Heading hierarchy is logical (h1 → h2 → h3, no skipping)
  • Only one <h1> and one <main> per page
  • All images have descriptive alt text (decorative images use alt="")
  • All images have explicit width and height attributes
  • SVG icons have aria-hidden="true" or appropriate labels

Accessibility

  • Skip link as first focusable element: <a href="#main">Skip to content</a>
  • All form inputs have associated <label>
  • All interactive elements reachable by keyboard
  • Focus styles visible (not just outline: none)
  • Color contrast ≥ 4.5:1 for normal text, ≥ 3:1 for large text and UI
  • No information conveyed by color alone
  • Modals trap focus and restore on close
  • Dynamic content changes announced via ARIA live regions
  • All <nav> elements have unique aria-label when multiple exist
  • Current page in navigation marked with aria-current="page"
  • Error messages programmatically associated with form fields
  • Never used tabindex greater than 0

SEO & Meta

  • Unique <title> (50-60 chars) on every page
  • Unique meta description (150-160 chars) on every page
  • link rel="canonical" on every page
  • Open Graph tags on all shareable pages
  • Twitter Card tags present
  • JSON-LD structured data appropriate for content type
  • Favicon in SVG + PNG + Apple touch icon
  • robots.txt and sitemap.xml present
  • Hreflang for multilingual sites

Performance

  • LCP image: loading="eager", fetchpriority="high", preloaded
  • All below-fold images: loading="lazy"
  • All images: decoding="async"
  • Images served in AVIF/WebP with fallback via <picture>
  • Responsive images use srcset and sizes
  • Videos serve multiple codec sources (AV1, VP9, H.264)
  • Animated GIFs replaced with <video>
  • Scripts use defer or async; never render-blocking in <head>
  • Critical CSS inlined; non-critical CSS loaded asynchronously
  • Fonts preloaded and use font-display: swap
  • Preconnect to critical third-party origins
  • HTTP/2 or HTTP/3 enabled
  • Brotli/gzip compression enabled
  • Cache headers set appropriately
  • LCP < 2.5s, INP < 200ms, CLS < 0.1 — verified with field data

CSS

  • No float-based layouts (use Grid/Flexbox)
  • No CSS @import in production
  • No unnecessary vendor prefixes for modern properties
  • box-sizing: border-box applied globally
  • Focus styles use :focus-visible, not removed
  • Animation properties use transform/opacity (not layout properties)
  • Rem/em for font sizes (respects user preferences)
  • No !important abuse

JavaScript

  • No document.write()
  • No escape()/unescape() — use encodeURIComponent()
  • Scroll events replaced with IntersectionObserver where appropriate
  • Resize events replaced with ResizeObserver where appropriate
  • DOM polling replaced with MutationObserver
  • async/await used instead of deep promise chains
  • Event listeners cleaned up to prevent memory leaks
  • Large data processing uses Web Workers or scheduler.yield()
